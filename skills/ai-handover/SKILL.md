  包含两个核心功能：

  支持三种存储模式：
  - A: 当前对话所在文件夹（子需求场景）
  - B: 项目根目录（推荐）
  - C: 用户自定义路径

# AI 工作交接套装 (ai-handover)

## 触发条件

| 命令 | 功能 |
|-----|------|
| `/ai-handover` | 显示功能菜单，选择生成交接或接手工作 |
| `/gzjj` | 直接生成交接文档 |
| `/xrtk` | 直接接手工作 |
| "工作交接"、"生成交接文档" | 生成交接文档 |
| "新人填坑"、"接手工作"、"接手" | 接手工作 |

## 功能一：生成交接文档 (gzjj)

### 工作流程

#### Step 1: 检查持久化配置

读取配置文件 `.ai-handover-config.json`，路径优先级：
1. 当前工作目录下的 `.ai-handover-config.json`
2. 用户全局目录 `~/.claude/.ai-handover-config.json`

配置文件结构：
```json
{
  "storageMode": "B",
  "basePath": "D:/project",
  "storagePath": "D:/project/工作交接文档-skill存储目录",
  "lastProjectRoot": "D:/project",
  "createdAt": "2026-04-23T10:00:00Z"
}
```

#### Step 2: 首次使用或项目变更时询问存储位置

**触发询问的条件**：
- 配置文件不存在
- 配置存在但当前项目根目录与 lastProjectRoot 不同

**使用 AskUserQuestion 询问**：

```
问题：请选择工作交接文档的基础存储位置

选项：
A. 当前对话所在文件夹
   → 将在当前文件夹下创建"工作交接文档-skill存储目录"子文件夹
   → 适合大项目中的子需求场景

B. 项目根目录（推荐）
   → 将在项目根目录下创建"工作交接文档-skill存储目录"子文件夹
   → 推荐：集中管理所有交接文档

C. 自定义文件夹路径
   → 将在指定路径下创建"工作交接文档-skill存储目录"子文件夹
```

#### Step 3: 确定存储路径并计算编号

| 选项 | 基础路径 | 最终存储路径 | 持久化 |
|-----|---------|-------------|-------|
| A | 当前对话所在文件夹 | `{当前文件夹}/工作交接文档-skill存储目录/` | 不持久化 |
| B | 项目根目录 | `{项目根目录}/工作交接文档-skill存储目录/` | 持久化 |
| C | 用户输入的路径 | `{用户路径}/工作交接文档-skill存储目录/` | 持久化 |

**项目根目录识别方法**：
1. 向上查找包含 `.git/`、`package.json`、`CLAUDE.md` 的目录
2. 若找不到，使用当前工作目录

**编号计算逻辑**：
```
1. 遍历存储文件夹内所有文件
2. 匹配 time_{数字}_工作交接_ 格式
3. 找出最大编号 max_x
4. 本次编号 x = max_x + 1（如果文件夹为空，x = 1）
```

**文件命名格式**：
```
time_{x}_基于{入口文件名}的{功能简述}交接文档-{YYYYMMDD-HHmmss}.md
```

**示例**：
```
time_1_基于index.html的OCR简谱识别交接文档-20260423-153045.md
time_2_基于editor.html的简谱编辑器交接文档-20260423-160000.md
```

#### Step 4: 显示确认信息

向用户显示：
```
📁 文档将存储至：
   {完整存储路径}

   这将是第 {x} 次工作总结
   文件名：time_{x}_基于{入口文件}的{功能简述}交接文档-{时间戳}.md

确认存储？
```

使用 AskUserQuestion 让用户确认。

#### Step 5: ⭐ 关键步骤 - 状态验证

**在生成交接文档前，必须验证关键状态**，区分"用户声称"和"实际完成"。

**从对话中提取关键事项清单**：

| 事项类型 | 示例 | 验证方法 |
|---------|------|---------|
| 代码开发 | "创建了 server-local.js" | 检查文件是否存在 |
| 配置完成 | "Supabase SQL 已执行" | 询问用户确认 |
| 部署完成 | "已部署到 Vercel" | 询问用户确认 / 检查 git 状态 |
| 测试通过 | "本地测试通过" | 查看对话中是否有测试结果 |

**自动验证**（可直接检查）：
- 检查文件是否存在
- 检查 git 状态
- 检查最近的提交

**需要用户确认的事项**（无法自动验证）：
- 云服务配置（Supabase/Vercel 等）
- 账号注册/充值
- 外部依赖安装
- 线上部署状态

使用 AskUserQuestion 让用户确认无法自动验证的事项。

#### Step 6: 状态标记规范

| 标记 | 含义 | 使用场景 |
|-----|------|---------|
| ✅ | 已验证通过 | 文件存在 / 测试通过 |
| ✅(用户确认) | 用户声称完成，未自动验证 | 云服务配置等 |
| 🔄 | 进行中 | 部分完成 |
| ⏳ | 待开始 | 尚未开始 |
| ❓ | 状态不明 | 需接手者验证 |
| ❌ | 已失败 | 确认失败 |

#### Step 7: 确定核心入口文件

**入口文件优先级**：

| 优先级 | 文件类型 | 示例 |
|-------|---------|------|
| 1 | 入口页面 | `index.html`, `app.html` |
| 2 | 前端代码 | `main.js`, `app.js`, `App.vue` |
| 3 | 后端代码 | `server.js`, `app.py`, `main.go` |

**确定方法**：
1. 从对话上下文提取用户工作最多的文件
2. 按优先级扫描
3. 取相对路径的文件名

#### Step 8: 生成工作交接文档

**⚠️ 路径规范（强制）**：
- 所有文件路径必须使用**绝对路径**，以项目根目录为基准
- 格式：`{项目根目录}/相对路径`
- ❌ 禁止：相对路径单独使用

## 功能二：接手工作 (xrtk)

### 工作流程

#### Step 1: 发现交接文档

- 直接读取该文档，跳过遍历

**如果用户未指定**：按优先级遍历

| 优先级 | 路径 |

**遍历规则**：
- 文件名匹配：`time_{数字}_*.md`

**从用户选择的工作交接文档名提取编号**：

## 双链结构说明

### 什么是 Obsidian 双链

使用 `[[]]` 语法链接文档：
- `[[文件名]]` 创建一个链接
- 在被链接文档中可以查看"反向链接"

### 双链的作用

```
工作交接文档                    新人填坑文档
time_3_工作交接_xxx.md          time_3_新人填坑_xxx.md
      │                              │
      │  [[新人填坑文档]]             │  [[工作交接文档]]
      │◄─────────────────────────────│
      │                              │
      │  被引用记录                    │  关联文档
      │  追加到这里                    │  记录在这里
```

### 双链维护规则

1. **生成新填坑文档时**：
   - 在新文档中添加：`[[交接文档名]]`
   - 在交接文档末尾追加：被引用记录

2. **多个填坑文档引用同一交接文档**：
   - 交接文档的"被引用记录"会列出所有引用
   - 形成文档网络，方便追溯

## 输出风格指南

### 核心原则

1. **简洁**：不说废话，直奔主题
2. **结构化**：理解 → 想法 → 选择，层次清晰
3. **弹框交互**：使用 AskUserQuestion，让用户点点选
4. **留痕**：生成文档，双链关联

### 禁止

- 心理咨询腔（"我理解你的感受"）
- 过度包装（"让我们一起来探索"）
- 纯文本让用户手打回复（应该用选择框）

## 方法论参考

### 工作交接核心原则

1. **准确性**：区分"用户声称"和"实际完成"，避免误导接手者
2. **完整性**：不遗漏关键信息，接手者无需追问
3. **结构化**：信息分层清晰，快速定位
4. **可执行性**：下一步行动明确，可直接开始
5. **上下文**：包含决策理由，避免重复试错

### 交接文档必备要素

| 要素 | 说明 | 重要性 | 验证要求 |
|-----|------|-------|---------|
| 当前状态 | 做到哪一步了 | ⭐⭐⭐ | 必须验证 |
| 已完成工作 | 做了什么 | ⭐⭐⭐ | 尽量验证 |
| 待完成工作 | 还要做什么 | ⭐⭐⭐ | - |
| 卡点阻碍 | 为什么停了 | ⭐⭐⭐ | - |
